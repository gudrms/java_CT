#### Q1. "일 8,000명, 하루 3-4만 건 처리하는 대용량 시스템을 어떻게 설계했나요?"

**예상 포인트**: 대용량 트래픽 처리 경험 검증

**답변 준비**:
```
[상황]
- 하루 8,000명 접수, 1인당 평균 4-5개 동의서 작성
- 피크 타임(오전 9-11시) 동시 접속자 500명 이상 집중

[설계 전략]
1. Redis 캐싱 전략
   - 자주 조회되는 동의서 템플릿을 Redis에 캐싱
   - TTL 설정으로 메모리 효율 관리
   - 조회 성능 3초 → 0.5초 (80% 개선)

2. 데이터베이스 최적화
   - 실행 계획(Execution Plan) 분석으로 인덱스 재설계
   - 대량 INSERT 시 Batch 처리로 트랜잭션 묶음 처리
   - 파티셔닝으로 날짜별 데이터 분산

3. 부하 분산
   - L4 로드밸런서 구성으로 2대 WAS 분산
   - 정적 리소스는 CDN 활용

[결과]
- 피크 타임 응답시간 3초 → 0.5초
- 서버 다운 없이 안정적 운영
```

**실무 팁**: 숫자로 말하기 (처리량, 응답시간, 개선율)

#### Q2. "Redis 캐싱 전략을 구체적으로 어떻게 적용했나요?"

**예상 포인트**: Redis 실무 경험 검증

**답변 준비**:
```
[캐싱 대상 선정]
1. 동의서 템플릿 (변경 빈도 낮음, 조회 빈도 높음)
2. 사용자 세션 정보
3. 자주 조회되는 검진 코드 매핑

[캐싱 전략]
- Cache-Aside 패턴 사용
  1. 요청 → Redis 조회
  2. 없으면 → DB 조회 후 Redis 저장
  3. TTL 1시간 설정 (동의서 템플릿 기준)

- Key 설계: "consent:template:{id}"
- Serialization: JSON 형태로 저장

[장애 대응]
- Redis 다운 시에도 DB fallback으로 서비스 유지
- Redis 커넥션 풀 설정 (최소 10, 최대 50)
```

**추가 질문 대비**:
- "Redis가 다운되면 어떻게 되나요?" → Fallback 로직 설명
- "캐시 무효화는 어떻게 처리하나요?" → 템플릿 수정 시 해당 Key 삭제

#### Q3. "쿼리 최적화를 어떻게 했나요? 구체적인 예시는?"

**예상 포인트**: SQL 튜닝 실력 검증

**답변 준비**:
```
[문제 상황]
동의서 목록 조회 쿼리가 3초 이상 소요 (전체 스캔 발생)

[분석]
1. Execution Plan 확인
   - WHERE 절의 create_date 컬럼에 인덱스 없음
   - JOIN 시 드라이빙 테이블 선택 비효율

2. 실행 계획 분석 결과
   - Full Table Scan 발생 (COST: 15000)
   - 불필요한 서브쿼리로 중복 조회

[개선]
1. 인덱스 추가
   CREATE INDEX idx_consent_create_date ON consent(create_date, user_id);

2. 쿼리 리팩토링
   - 서브쿼리 → JOIN으로 변경
   - SELECT 컬럼 최소화 (필요한 것만)
   - LIMIT/OFFSET 대신 커서 기반 페이징

[결과]
- 3초 → 0.5초 (83% 개선)
- COST: 15000 → 300
```

**실무 팁**: "EXPLAIN PLAN으로 분석했다"고 말하면 신뢰도 UP

2. 팀 분리 필요성
   - 식권팀, 동의서팀 독립 개발 어려움

[분리 전략]
Domain-Driven Design 기반 분리
1. 식권 서비스 (Meal Service)
   - QR 발급/검증, 정산 배치

2. 전자동의서 서비스 (Consent Service)
   - 동의서 CRUD, PDF 생성, 전자서명

3. 문진표 서비스 (Survey Service)
   - 동적 폼, 응답 관리

[기술 스택]
- API Gateway: Spring Cloud Gateway
- Service Discovery: 없음 (직접 호출)
- 통신: REST API (동기), RabbitMQ (비동기)
- Database: 서비스별 독립 DB

[어려웠던 점]
- 트랜잭션 분산 문제 → Saga 패턴 (보상 트랜잭션)

**추가 질문 대비**:
- "서비스 간 트랜잭션은 어떻게 처리하나요?" → Saga 패턴 설명
- "MSA의 단점은 없었나요?" → 복잡도 증가, 모니터링 어려움

#### Q8. "MyBatis vs JPA, 왜 MyBatis를 선택했나요?"

**답변 준비**:
```
[MyBatis 선택 이유]
1. 복잡한 쿼리
   - 다중 JOIN, 서브쿼리가 많은 레거시 DB
   - XML로 쿼리 명시적 관리 선호

2. 성능 튜닝
   - 직접 쿼리 작성으로 최적화 가능
   - 동적 쿼리 (if, choose) 활용

3. 기존 시스템 호환
   - 레거시 Oracle DB 구조 그대로 활용

[JPA 장단점 인지]
장점: 생산성, 객체 지향적
단점: 복잡한 쿼리 시 Native Query 필요, N+1 문제

[향후 방향]
- 신규 프로젝트는 Spring Data JDBC 고려
- 간단한 CRUD는 JPA, 복잡한 쿼리는 MyBatis 혼용
```

## 4️⃣ 상황 대응 질문

#### Q12. "장애 상황을 경험한 적 있나요? 어떻게 대응했나요?"

**답변 준비 (STAR 기법)**:
```
[Situation]
전자동의서 시스템에서 피크 타임(오전 9시) DB 커넥션 풀 고갈 발생
→ 사용자 500명 이상 동시 접속 시 "Connection timeout" 에러

[Task]
즉시 서비스 복구 및 근본 원인 파악

[Action]
1. 긴급 대응 (5분)
   - HikariCP 커넥션 풀 크기 20 → 50으로 증설
   - 서버 재시작 없이 설정 reload

2. 원인 분석 (1시간)
   - 슬로우 쿼리 로그 확인
   - 동의서 목록 조회 쿼리가 3초 소요 (인덱스 없음)
   - 커넥션 반환 지연 발생

3. 근본 해결 (1일)
   - 인덱스 추가로 쿼리 0.5초 단축
   - Redis 캐싱 도입
   - 커넥션 leak 방지 코드 추가 (try-with-resources)

[Result]
- 장애 시간 10분 이내 복구
- 이후 동일 장애 발생 없음
- 커넥션 풀 사용률 80% → 40%로 안정화
```

[Result]
